## 三、成功率分析

| 检查项 | 通过 | 失败 | 通过率 |
|--------|------|------|--------|
| status 302/404/403 | 15,169 | 765,155 | **1.94%** |
| latency < 50ms | 672,191 | 108,133 | 86.14% |
| latency < 20ms | 420,155 | 360,169 | 53.84% |

**成功率仅 1.94% 的根因: Gateway 限流器**

Gateway 配置了 IP 级限流: `100 req/s, burst 200`。压测从单一 IP 发起，800 VU 并发远超限流阈值，导致 **98% 的请求被返回 HTTP 429** (Too Many Requests)。

这不是系统 bug — 限流器在正常工作！但从压测角度看，它掩盖了后端的真实性能。

### 限流器下的延迟特征

- 429 响应极快 (~0.2ms)，拉低了整体 P50 到 18ms
- 302 成功响应的真实 P50 = 49.63ms (2.7 倍于整体 P50)
- 说明 **MySQL 直查路径在并发压力下延迟显著上升**

## 四、瓶颈分析

### 4.1 Gateway 限流器 (已确认)

```
当前配置: 100 req/s per IP, burst 200
压测流量: 5,201 req/s from single IP
限流比例: 98%
```

**问题**: 限流器的 `sync.Map` 存储所有 IP bucket 且无 TTL 清理机制，长时间运行内存持续增长。此外，限流阈值对单 IP 来说过于严格。

### 4.2 热路径无 Redis 缓存 (已确认)

成功 302 响应的延迟:
- P50 = 49.63ms — 每次都查 MySQL，无缓存
- P95 = 346ms — MySQL 在高并发下 buffer pool 压力大

对比: 如果有 Redis 缓存，P50 应该 < 2ms (Redis GET 约 0.5ms)。

### 4.3 Gateway Proxy 复用问题 (已确认)

每次请求都 `url.Parse()` + `NewSingleHostReverseProxy()`，800 VU 下每秒创建 5000+ 个 proxy 对象，GC 压力导致 P95 尾部延迟飙升到 80ms+。

### 4.4 GORM 未配置连接池 (已确认)

默认 `MaxOpenConns = 0` (无限)，800 VU 并发下可能打开了数千个 MySQL 连接，MySQL `max_connections` 默认 151，超出后会导致 `too many connections`。
