## 三、分析

### Round 1 → Round 2: 后端优化生效 ✅

- 302 P50: 49ms → 32ms (**-35%**)
- 302 P95: 346ms → 154ms (**-56%**)
- 302 最大: 1,957ms → 526ms (**-73%**)
- 后端延迟降低 35-73%，Redis 缓存效果显著

### Round 2 → Round 3: 限流放行 + 后端瓶颈暴露 ⚠️

- 限流从 100 → 1000 r/s，302 成功数从 15K → 120K (8x)
- 但 1200 VU 并发让后端严重过载:
  - 302 P50: 32ms → 221ms (后端瓶颈)
  - 302 P95: 154ms → 2,006ms (严重过载)
- **根因**: Link 服务单实例在高并发下成为瓶颈

### 为什么 Round 3 的 overall QPS 反而降低了？

```
Round 1/2: 98% 请求被 100r/s 限流 → 429 响应 (0.2ms) → overall QPS 看起来很高
Round 3:   55% 请求通过限流 → 真正打到 Link 服务 → 处理时间变长 → overall QPS 降低
```

这不是退化 — 这是**真实负载暴露**。之前 98% 的请求被限流器拦住了，
现在更多请求到达后端，后端处理不过来导致延迟上升和超时。

## 四、结论 & 下一步

### ✅ 已确认有效的优化

| 优化 | 效果 |
|------|------|
| Redis 缓存 | 单请求 4x 加速 (20ms → 5ms) |
| Gateway Proxy 复用 | 消除 GC 对象分配 |
| GORM 连接池 | 防止连接耗尽 |
| 限流阈值 1000 | 302 成功数提升 8 倍 |
| 缓存预热 | 启动即命中，消除冷启动 |
| gin.New() | 去掉 Logger I/O 开销 |
| Kafka BatchTimeout | 10ns → 100ms，批量发送 |

### ⚠️ 当前瓶颈: Link 服务单实例

1200 VU 压测暴露了真正的瓶颈 — **Link 服务单实例处理能力不足**。
当更多请求穿透限流到达后端时，单实例 MySQL 查询 + Go runtime 并发处理能力成为天花板。

### 🔜 下一步优化方向

| 方向 | 操作 | 预期效果 |
|------|------|----------|
| **水平扩展** | 多实例 Link 服务 + Nginx 负载均衡 | QPS 线性扩展 |
| **连接池优化** | 增大 MaxOpenConns 到 200+ | 提升并发处理 |
| **fasthttp 替代** | Gateway 用 fasthttp 反代 | 吞吐提升 2-3x |
| **读写分离** | 热路径走 MySQL 从库 | 降低主库压力 |
