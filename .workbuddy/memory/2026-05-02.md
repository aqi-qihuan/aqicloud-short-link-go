## 性能分析与压测方案

对 aqicloud-short-link-go 项目进行了全面的性能分析，识别出 9 个性能瓶颈：

### P0 (致命级)
1. **热路径无 Redis 缓存** — `link_api.go:50` 每次短链重定向都查 MySQL，这是性能差的根本原因
2. **Gateway 每次请求新建 Proxy** — `gateway/main.go:42-51` 每次都 url.Parse + NewSingleHostReverseProxy，GC 压力巨大

### P1 (严重级)
3. **GORM 未配置连接池** — `link/main.go:42` 无 MaxOpenConns 设置
4. **限流器内存泄漏** — `ratelimit.go:50` sync.Map 无 TTL 清理
5. **RabbitMQ 单 Channel** — `rabbitmq.go:17` 整个应用只有 1 个 Channel
6. **Kafka BatchTimeout 值错误** — `kafka.go:22` `BatchTimeout: 10` 实际是 10ns 而非 10ms

### P2 (中等级)
7. MQ 消费者单协程
8. MQ 发布未确认 (Publisher Confirms)
9. Gateway 使用 gin.Default() 多余日志

## Git 工作流优化

对项目进行了完整的 Git 工作流建设：

### 发现的问题
- 项目未初始化 Git 仓库
- .gitignore 不完整（缺少编译产物、密钥目录忽略）
- RSA 密钥文件在项目目录中（已被 .gitignore 排除）
- 无 Git Hooks、无 CI/CD、无团队协作规范

### 实施的改进
1. **Git 仓库初始化** — git init + 完善 .gitignore（编译产物、密钥、IDE 文件等）
2. **CONTRIBUTING.md** — 完整的团队协作规范（Git Flow 简化版分支策略、Conventional Commits 提交规范、PR 流程、合并策略、版本发布流程）
3. **Git Hooks** — `scripts/githooks/pre-commit`（gofmt/go vet/golangci-lint/敏感信息检查）+ `scripts/githooks/commit-msg`（Conventional Commits 格式校验）
4. **CI 工作流** — `.github/workflows/ci.yml`（lint → test → build 三阶段流水线）
5. **代码质量配置** — `.golangci.yml`（errcheck/gosimple/govet/staticcheck 等）
6. **增强 Makefile** — 添加 fmt/vet/install-hooks/docker-build/test-cover/help 等命令
7. **PR 模板 + Issue 模板** — `.github/pull_request_template.md`、`.github/ISSUE_TEMPLATE/`
8. **分支策略** — 创建了 main + develop 双分支结构
